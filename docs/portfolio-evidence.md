# Part A. 프로젝트 개요

## 1. 프로젝트 한 줄 소개
연구논문(한국회계학회지 게재)이 제안한 PRS 통합 공시 지침을, 논문 원문과 항목 단위로 추적되는 평가 루브릭과 재사용 가능한 다중 에이전트 개발 체계로 구조화하는 프로젝트.

## 2. 프로젝트 배경과 문제 정의
국내 PRS(주가수익스와프) 계약 규모가 2022년 약 992억 원에서 2025년 9월 10조 8,598억 원으로 급증했으나, 기업마다 공시 항목·명칭·상세도가 달라 투자자가 거래의 경제적 실질(부채성 여부, 잠재적 현금유출)을 비교하기 어렵다. 기반 논문이 10개 기업의 실제 공시를 심층 분석해 이 문제를 실증했다.

## 3. 주요 사용자와 사용 시나리오
[미확정 — 원 기획서상 "기업명·평가기간을 입력하면 자동 채점" 시나리오를 상정하고 있으나, 실사용자 검증은 아직 없음]

## 4. 프로젝트 목표와 성공 기준
목표: 논문의 공시 가이드라인을 재현 가능·설명 가능한 채점 도구로 전환. 성공 기준은 배점이 아직 확정되지 않아 정량적으로 정의하지 못함(`_workspace/rubric_gaps.md` 참고).

## 5. 나의 역할과 기여
⚠️ **기반 논문 연구(최아리·고민서·박신화·이규진·현정훈, 공동저자 5인)와 PRSTI 소프트웨어화 작업(본인 단독 수행)을 구분합니다.** 아래는 소프트웨어화 작업에 한정된 기여입니다.
- 논문의 통합 공시 지침(19개 항목)을 평가 루브릭 후보로 전환
- 초기 기획안과 논문 원문을 대조해 근거 없는 채점 항목 3개를 발견·제외 (→ PE-001)
- 재사용 가능한 다중 에이전트 개발 체계(하네스) 설계
- Notion 포트폴리오 페이지 작성 및 유지
- **AI 협업 범위**: Claude Code(Claude)를 페어 개발 도구로 사용해 문서·설계 초안을 작성했으며, 논문 대조·항목 채택 여부 등 회계적·구조적 판단은 본인이 최종 결정. 어떤 판단을 AI가 제안하고 본인이 확정했는지는 `decision-log.md`의 "결정 주체" 필드에 매번 구분해 기록.

## 6. 회계적 판단과 기술적 구현을 연결한 과정
표6~8의 "필수/조건부필수/권장" 3단 분류(회계적 중요도 판단)를 rubric.yaml의 `tier` 필드로, "조건부 필수항목의 해당없음 인정"(회계적 판단)을 `na_rule` 필드로 구조화. 논문에 명시적 근거가 없는 항목은 시스템적으로 `excluded_candidates`로 분리해 회계적 엄밀성이 코드 구조에 반영되도록 설계.

## 8. 전체 시스템 아키텍처
현재 실재하는 것: 5개 에이전트(rubric-architect, dart-researcher, evidence-extractor, scoring-engine-dev, prsti-auditor) + 6개 스킬로 구성된 서브에이전트 파이프라인(`prsti-harness` 참고). 전체 서비스 아키텍처(DART~웹)는 설계 문서 수준으로만 존재.

## AI 활용 포트폴리오 요건 대조 (2026-07-20 추가)

회계법인 지원 시 "AI 활용"을 자기소개서의 2순위 강점으로 쓰려면 세 가지 요구사항과 "GitHub/Notion으로 확인 가능한 산출물"이 필요하다는 커리어 조언을 받았다. PRSTI 현재 상태를 그 기준에 정직하게 대조한다.

| 요구사항 | 상태 | 근거 | 언제 충족되는가 |
|---|---|---|---|
| Agent·Harness·Skills 등 최신 AI 기술 이해 | **충족** | 전용 에이전트 5개·스킬 6개를 실제로 설계. 매뉴얼(팀 모드) 대신 환경 제약을 스스로 확인하고 대안(서브에이전트)을 판단한 사례 있음(DEC-001) | 이미 충족(문서·설계 수준) |
| 생성형 AI API·오픈소스 LLM 활용 프로젝트 경험 | **미충족** | evidence-extractor(Phase 3)가 LLM API를 호출하도록 설계는 됐으나 코드 없음. "Claude Code로 개발했다"와 "AI API를 활용하는 제품을 만들었다"는 다른 주장이므로 혼동하지 않음(→ 5번 AI 협업 범위) | Phase 3 착수 후 |
| 프로그래밍을 통한 AI Product 개발 가능 | **미충족** | 실행 가능한 코드 전무(전부 [계획단계]/[설계완료·미구현], → 10번) | Phase 1(점수 엔진) 착수 후 |
| GitHub/Notion으로 확인 가능한 산출물 | **부분 충족** | Notion 페이지는 있으나 "서비스 미구현" 상태를 그대로 노출 중(정직성 원칙상 의도된 것). GitHub은 이 프로젝트에 아예 없음 — git 자체가 미도입(TS-003, DEC-004) | GitHub 저장소 개설은 git 설치·원격 공개 범위 결정이 필요해 사용자 확인 없이 진행하지 않음 |

**종합**: 지금 시점에 "AI 활용"을 2순위로 내세우기엔 이르다. Agent/Harness/Skills 이해는 이미 보여줄 수 있지만, "실제 AI Product 개발 경험"과 "생성형 AI API 활용 경험"은 Phase 1·3이 실제 코드로 존재해야 채워진다. 지금 자기소개서에 쓴다면 "AI에 관심 있음"이 아니라 정확히 "다중 에이전트 개발 체계를 설계·구축하는 중"이라는 현재 상태로 표현해야 한다.

# Part B. STAR 사례 모음

## PE-001: 논문 근거 없는 채점 항목 발견과 제외

1. **해결하려던 문제**: 초기 기획안의 5영역 평가체계가 실제로 논문 원문에서 도출된 것인지 검증되지 않은 상태였음.
2. **내가 수행한 구체적 행동**: 기획안의 5영역·하위항목을 논문 표6(필수 12)·표7(조건부필수 3)·표8(권장 4) 19개 항목과 하나씩 대조. "PRS 명칭"·"공시 위치"·"진성매각 판단" 3개가 논문에 채점 항목으로 존재하지 않음을 확인.
3. **사용한 기술과 선택 이유**: 논문 원문 재독 + 항목별 출처 추적표 작성(rubric-architect 절차의 핵심 방법론)
4. **검토한 대안과 최종 의사결정**: → DEC-002 (논문 원 카테고리 잠정 채택, 5영역안 보류)
5. **테스트 및 검증 방법**: 19개 항목 전체가 `source_table`/`source_item_id`를 갖는지 확인(자체 검증 가능)
6. **정량적 결과**: 19/19 항목 매핑(100%), 미출처 후보 3건 별도 분리
7. **실패 또는 문제 해결 과정**: 해당 없음(이 사례 자체가 문제 발견·해결 과정)
8. **배운 점**: "논문을 참고했다"는 것과 "논문에서 도출했다"는 것은 다르며, 이 차이를 항목 단위로 검증하지 않으면 지표의 학술적 근거가 조용히 흐려질 수 있음을 확인.
9. **회계·감사·IT 직무와의 연결**: 근거 없는 항목을 걸러내는 절차는 감사에서 "충분하고 적합한 감사증거" 개념과 본질적으로 같은 문제의식.
10. **STAR 소재**:
    - S: 기획안의 평가체계가 논문에서 실제로 도출됐는지 검증되지 않은 상태였다.
    - T: 19개 항목 전체를 논문 원문과 1:1로 대조해 출처를 확인해야 했다.
    - A: 표6~8을 항목 단위로 재추출하고, 기획안의 하위 항목과 대조해 불일치 3건을 발견했다.
    - R: 근거 없는 항목을 조용히 포함시키는 대신 별도 목록으로 분리하고, 논문 원 카테고리를 잠정 채택해 추적성을 확보했다.
