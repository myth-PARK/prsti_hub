## 1. 프로젝트 한 줄 소개
연구논문(한국회계학회지 게재)이 제안한 PRS 통합 공시 지침을 추적 가능한 평가 루브릭으로 구조화하고, 생성형 AI가 공시 원문에서 근거를 추출하도록 설계한 다중 에이전트 개발 체계로 구현하는 프로젝트.

## 2. 프로젝트 배경과 문제 정의
국내 PRS(주가수익스와프) 계약 규모가 2022년 약 992억 원에서 2025년 9월 10조 8,598억 원으로 급증했으나, 기업마다 공시 항목·명칭·상세도가 달라 투자자가 거래의 경제적 실질(부채성 여부, 잠재적 현금유출)을 비교하기 어렵다. 기반 논문이 10개 기업의 실제 공시를 심층 분석해 이 문제를 실증했다.

## 5. 나의 역할과 기여
⚠️ **기반 논문 연구(최아리·고민서·박신화·이규진·현정훈, 공동저자 5인)와 PRSTI 소프트웨어화 작업(본인 단독 수행)을 구분합니다.** 아래는 소프트웨어화 작업에 한정된 기여입니다.
- 논문의 통합 공시 지침(19개 항목)을 평가 루브릭 후보로 전환
- 초기 기획안과 논문 원문을 대조해 근거 없는 채점 항목 3개를 발견·제외 (→ PE-001)
- 재사용 가능한 다중 에이전트 개발 체계(하네스) 설계
- Notion 포트폴리오 페이지 작성 및 유지
- **AI 협업 범위**: Claude Code(Claude)를 페어 개발 도구로 사용해 문서·설계 초안을 작성했으며, 논문 대조·항목 채택 여부 등 회계적·구조적 판단은 본인이 최종 결정. 어떤 판단을 AI가 제안하고 본인이 확정했는지는 `decision-log.md`의 "결정 주체" 필드에 매번 구분해 기록.

## 6. 회계적 판단과 기술적 구현을 연결한 과정

현재 실재하는 것: (1) 5개 에이전트(rubric-architect, dart-researcher, evidence-extractor, scoring-engine-dev, prsti-auditor) + 6개 스킬로 구성된 서브에이전트 개발 절차(`prsti-harness` 참고), (2) 실제 동작하는 파이썬 코드 3개 패키지 — `prsti_common`(rubric.yaml 공유 로더), `evidence_extractor`(Claude API 구조화 출력 클라이언트), `scoring_engine`(결정론적 규칙 엔진). 전체 서비스 아키텍처(DART~웹)는 여전히 설계 문서 수준.

## 9. 데이터 수집·처리·평가 흐름
논문 표6~8 → 루브릭(v1.0, 승인 완료) → (계획) DART 소량 조회 → AI 근거 추출(제안값, `evidence_extractor` — 코드 완성·라이브 미검증) → 사용자 검토(확정값) → 결정론적 점수 계산(`scoring_engine` — 코드 완성·실행 검증 완료). 뒷단(점수 계산)은 실제 데이터로 여러 시나리오를 통과시켜 검증했고, 앞단(DART 수집)은 아직 실제 데이터가 흐른 적 없음.

## 12. 테스트·검증 방법
`pytest` 기반 자동 테스트 23개(전부 통과): `evidence_extractor` 12개(루브릭 로더·프롬프트 구성·환각 검증 로직은 실제 실행, Claude API 호출부만 `unittest.mock`으로 배선 검증), `scoring_engine` 10개(API를 쓰지 않는 순수 로직이라 mock 없이 전부 실제 실행 — 결정론성·조건부 na_rule 처리·권장항목 미반영·`suggested_score` 미사용을 각각 별도 테스트로 강제), 루브릭 구조 1개. 여기에 더해 `prsti-qa-checklist`의 수동 절차(루브릭↔논문 추적성 대조 등)를 병행 적용 중.

## 13. 정량적 결과와 실제 확인된 성과
- 논문 통합 공시 항목 매핑률: **19/19 (100%)** — `rubric/rubric.yaml`의 각 item에 `source_table`/`source_item_id`가 있는지 세어 검증 가능
- 자동 테스트: **23/23 통과** (`evidence_extractor` 12, `scoring_engine` 10, 로더 1)
- `scoring_engine`은 전체 100점 만점 시나리오(전 항목 만점/0점/조건부 auto_max/미검토 혼재)를 실제로 계산시켜 기대값과 일치함을 확인

## 15. 회계·감사·IT 직무와의 연결
논문의 정성적 서술(회계처리 판단, 위험과 보상의 이전 논의)을 근거 문장 단위로 추적 가능한 평가항목으로 옮기는 작업은, 감사 절차에서 "어떤 근거로 이 판단에 도달했는가"를 문서화하는 것과 같은 성격의 작업. AI 제안과 사람의 확정 판단을 항상 분리해 기록하는 설계는 감사조서에서 준비자·검토자 판단을 구분하는 방식과 같은 문제의식에서 나옴.
**AI 활용 관점**: evidence-extractor의 설계(생성형 AI가 원문에서 근거를 뽑되, 인용 없는 판정과 환각을 구조적으로 차단)는, 회계법인들이 도입 중인 AI 기반 감사 플랫폼(대규모 거래 데이터에서 이상 징후·근거를 추출하되 최종 판단은 감사인이 확정하는 구조)과 같은 문제의식을 소규모로 직접 설계해본 경험이다.

## PE-001: 논문 근거 없는 채점 항목 발견과 제외

1. **해결하려던 문제**: 초기 기획안의 5영역 평가체계가 실제로 논문 원문에서 도출된 것인지 검증되지 않은 상태였음.
2. **내가 수행한 구체적 행동**: 기획안의 5영역·하위항목을 논문 표6(필수 12)·표7(조건부필수 3)·표8(권장 4) 19개 항목과 하나씩 대조. "PRS 명칭"·"공시 위치"·"진성매각 판단" 3개가 논문에 채점 항목으로 존재하지 않음을 확인.
3. **사용한 기술과 선택 이유**: 논문 원문 재독 + 항목별 출처 추적표 작성(rubric-architect 절차의 핵심 방법론)
4. **검토한 대안과 최종 의사결정**: → DEC-002 (논문 원 카테고리 잠정 채택, 5영역안 보류)
5. **테스트 및 검증 방법**: 19개 항목 전체가 `source_table`/`source_item_id`를 갖는지 확인(자체 검증 가능)
6. **정량적 결과**: 19/19 항목 매핑(100%), 미출처 후보 3건 별도 분리
7. **실패 또는 문제 해결 과정**: 해당 없음(이 사례 자체가 문제 발견·해결 과정)
8. **배운 점**: "논문을 참고했다"는 것과 "논문에서 도출했다"는 것은 다르며, 이 차이를 항목 단위로 검증하지 않으면 지표의 학술적 근거가 조용히 흐려질 수 있음을 확인.
9. **회계·감사·IT 직무와의 연결**: 근거 없는 항목을 걸러내는 절차는 감사에서 "충분하고 적합한 감사증거" 개념과 본질적으로 같은 문제의식.
10. **STAR 소재**:
    - S: 기획안의 평가체계가 논문에서 실제로 도출됐는지 검증되지 않은 상태였다.
    - T: 19개 항목 전체를 논문 원문과 1:1로 대조해 출처를 확인해야 했다.
    - A: 표6~8을 항목 단위로 재추출하고, 기획안의 하위 항목과 대조해 불일치 3건을 발견했다.
    - R: 근거 없는 항목을 조용히 포함시키는 대신 별도 목록으로 분리하고, 논문 원 카테고리를 잠정 채택해 추적성을 확보했다.

## PE-002: 생성형 AI 근거 추출 컴포넌트의 환각 방지 설계

1. **해결하려던 문제**: 공시 원문에서 채점 근거를 뽑는 역할을 생성형 AI에 맡기면, AI가 원문에 없는 내용을 그럴듯하게 지어내는 환각(hallucination)이 시스템 신뢰성을 가장 크게 위협함.
2. **내가 수행한 구체적 행동**: evidence-extractor 에이전트와 prs-evidence-schema 스킬을 설계하면서 (1) 모든 `suggested_score>0`에는 원문 그대로의 `quote` 필드를 강제 (2) `quote`가 실제 원문(raw_excerpt)에 부분 문자열로 존재하는지 검증하는 절차를 prsti-auditor에 별도로 부여 (3) 근거를 못 찾으면 `not_found`로만 처리하고 추정 금지 (4) 신뢰도(confidence)를 "판정에 대한 확신"과 "존재 자체에 대한 확신"으로 구분해 혼동을 막음.
3. **사용한 기술과 선택 이유**: 구조화된 JSON 스키마(자유 서술형 대신) — 자유 서술은 검증이 불가능하지만, 스키마로 강제하면 기계적으로 대조 검증이 가능해짐.
4. **검토한 대안과 최종 의사결정**: AI가 직접 최종 점수를 내는 안(단순하지만 위험) vs AI는 제안만 하고 사람이 확정하는 안(복잡하지만 통제 가능) 중 후자를 채택 — 원 프로젝트의 핵심 원칙(AI/점수 분리)과 직결.
5. **테스트 및 검증 방법**: (설계 단계) prsti-qa-checklist 3절에 "quote가 raw_excerpt의 부분 문자열인지" 대조하는 절차를 명문화. 실제 코드 작성 시 이 절차를 자동화된 테스트로 전환 예정.
6. **정량적 결과**: 아직 실제 데이터로 측정된 수치 없음(설계만 완료, 코드 없음) — 개발 착수 후 환각 발생률을 측정할 계획.
7. **실패 또는 문제 해결 과정**: 해당 없음(설계 단계 사례).
8. **배운 점**: 생성형 AI를 신뢰할 수 있게 쓰는 방법은 "더 똑똑한 모델을 쓰는 것"이 아니라, AI 출력이 시스템의 최종 권위를 가지지 못하도록 구조로 막고 검증 가능한 형태(원문 인용)로만 받는 것이라는 점.
9. **회계·감사·IT 직무와의 연결**: "증거 없는 판정을 허용하지 않는다"는 원칙은 감사증거의 충분성·적합성 개념과 직접 연결됨.
10. **STAR 소재**:
    - S: 생성형 AI로 공시 근거를 자동 추출하는 컴포넌트가 필요했으나, AI 환각이 신뢰성을 해칠 위험이 컸다.
    - T: AI가 존재하지 않는 근거를 지어내지 못하도록 구조적으로 막는 설계가 필요했다.
    - A: 원문 인용 강제, 인용문 대조 검증 절차 분리, not_found 명시, 신뢰도 이원화를 스키마 수준에서 설계했다.
    - R: 실제 코드 작성 전 단계에서부터 환각 방지가 시스템 설계에 내장된 근거 추출 파이프라인을 완성했다.
