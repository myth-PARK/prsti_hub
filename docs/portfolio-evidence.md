## 1. 프로젝트 한 줄 소개
연구논문(한국회계학회지 게재)이 제안한 PRS 통합 공시 지침을 추적 가능한 평가 루브릭으로 구조화하고, 생성형 AI가 공시 원문에서 근거를 추출하도록 설계한 다중 에이전트 개발 체계로 구현하는 프로젝트.

## 2. 프로젝트 배경과 문제 정의
국내 PRS(주가수익스와프) 계약 규모가 2022년 약 992억 원에서 2025년 9월 10조 8,598억 원으로 급증했으나, 기업마다 공시 항목·명칭·상세도가 달라 투자자가 거래의 경제적 실질(부채성 여부, 잠재적 현금유출)을 비교하기 어렵다. 기반 논문이 10개 기업의 실제 공시를 심층 분석해 이 문제를 실증했다.

## 5. 나의 역할과 기여
⚠️ **기반 논문 연구(최아리·고민서·박신화·이규진·현정훈, 공동저자 5인)와 PRSTI 소프트웨어화 작업(본인 단독 수행)을 구분합니다.** 아래는 소프트웨어화 작업에 한정된 기여입니다.
- 논문의 통합 공시 지침(19개 항목)을 평가 루브릭 후보로 전환
- 초기 기획안과 논문 원문을 대조해 근거 없는 채점 항목 3개를 발견·제외 (→ PE-001)
- 재사용 가능한 다중 에이전트 개발 체계(하네스) 설계
- Notion 포트폴리오 페이지 작성 및 유지
- **AI 협업 범위**: Claude Code(Claude)를 페어 개발 도구로 사용해 문서·설계 초안을 작성했으며, 논문 대조·항목 채택 여부 등 회계적·구조적 판단은 본인이 최종 결정. 어떤 판단을 AI가 제안하고 본인이 확정했는지는 `decision-log.md`의 "결정 주체" 필드에 매번 구분해 기록.

## 6. 회계적 판단과 기술적 구현을 연결한 과정
표6~8의 "필수/조건부필수/권장" 3단 분류(회계적 중요도 판단)를 rubric.yaml의 `tier` 필드로, "조건부 필수항목의 해당없음 인정"(회계적 판단)을 `na_rule` 필드로 구조화. 논문에 명시적 근거가 없는 항목은 시스템적으로 `excluded_candidates`로 분리해 회계적 엄밀성이 코드 구조에 반영되도록 설계.

## 8. 전체 시스템 아키텍처
현재 실재하는 것: (1) 5개 에이전트(rubric-architect, dart-researcher, evidence-extractor, scoring-engine-dev, prsti-auditor) + 6개 스킬로 구성된 서브에이전트 개발 절차(`prsti-harness` 참고), (2) 실제 동작하는 파이썬 코드 6개 패키지 — `prsti_common`(rubric.yaml 공유 로더·비용 통제 설정), `dart_researcher`(DART Open API 클라이언트, 실제 조회 검증됨), `evidence_extractor`/`anthropic_extractor`(Claude API 구조화 출력 클라이언트, 선택적 비교 실험용으로 격리), `rule_based_extractor`(무료 규칙 기반 자동 채점, **기본 실행 경로**), `scoring_engine`(결정론적 점수 계산), `webapp`(Streamlit 로컬 웹 애플리케이션 — 임의 기업명 입력 → DART 조회 → 자동 채점 요약 화면). **DART 조회부터 최종 점수 산출, 웹 화면 표시까지의 전 구간이 API 키 없이(DART_API_KEY만으로) 실제로 동작한다.**

## 9. 데이터 수집·처리·평가 흐름
논문 표6~8 → 루브릭(v1.0, 승인 완료) + `rubric_rules.yaml`(기계 실행 규칙, 2026-07-21 신규) → DART 소량 조회(`dart_researcher` — 무료, 실제 조회 검증됨) → 근거 탐지·채점(`rule_based_extractor` — 무료, 키워드·패턴 기반 자동 판정) → 결정론적 총점 계산(`scoring_engine` — 변경 없이 재사용). 이 전체 경로가 한화솔루션 1개 기업에 대해 실제로 끝까지 실행됐다(`_workspace/dart_한화솔루션_2025.json` → `_workspace/derived/scored_한화솔루션_2025.json`). 별도 경로로 `evidence_extractor`(AI 제안값) → 사람 확정 흐름도 코드는 완성돼 있으나, 무과금 원칙에 따라 이 경로의 라이브 API 호출은 보류 중이다.

**설계 결정 배경(2026-07-21)**: 유료 생성형 AI API가 없으면 프로그램을 실행할 수 없는 구조를 개선하기 위해, 논문에서 확립한 평가 기준과 실제 공시 사례를 명시적인 규칙으로 변환했다. 이를 통해 API 비용 없이 동일한 입력에 동일한 결과를 반환하고, 각 점수의 원문 근거와 판정 과정을 추적할 수 있는 자동 채점 구조로 변경했다(DEC-014). **구현 완료**: `dart_researcher`·`rule_based_extractor`·`scoring_engine` 전 구간이 실제 코드로 존재하고 한화솔루션 실 데이터로 실행됨. **검증 완료**: 자동 테스트 50개, 실제 DART 조회 1건. **진행 예정**: 여러 기업으로 검증 표본 확대, insufficient_evidence 9개 항목의 실사례 추가 확보, rubric.yaml accepted_example 2건 수정(DEC-015).

## 12. 테스트·검증 방법
`pytest` 기반 자동 테스트 **60개(전부 통과)**: `evidence_extractor` 공용부 8개(루브릭 로더·프롬프트·환각 검증, 실제 실행), `anthropic_extractor` 5개(Claude API 호출부, `unittest.mock`으로 배선 검증 — `ALLOW_PAID_API` 게이트가 기본값에서 호출 자체를 차단하는지가 최우선 검증 대상), `scoring_engine` 10개(API 미사용, 결정론성·조건부 na_rule·권장항목 미반영을 mock 없이 실제 실행으로 검증), `dart_researcher` 6개(동의어 탐지·과다탐지 방지, mock), `rule_based_extractor` 22개(규칙 로더·키워드/패턴 탐지·점수 판정·파이프라인 결정론성·영역별 점수 배분을 mock 없이 실제 실행으로 검증. 특히 "만기 연장"이 콜옵션 항목으로 오분류되지 않는지, insufficient_evidence 항목이 매치가 있어도 임의로 점수를 올리지 않는지, `rule_based_extractor`를 서브프로세스에서 단독 import해도 anthropic 모듈이 전혀 로드되지 않는지를 각각 별도 테스트로 강제), `webapp` 9개(DART 조회 예외를 화면 오류 메시지로 안전하게 변환하는지 — `MissingAPIKeyError`·`DartApiError`·예상 못한 예외 3가지 경로 —, `streamlit.testing.v1.AppTest`로 페이지가 예외 없이 렌더링되는지, 기업명 입력→분석 버튼 클릭까지 실제 위젯 상호작용을 재현해도 문제없는지, 그리고 `streamlit run`이 실제로 사용하는 sys.path 조건(스크립트 폴더만, 저장소 루트 제외)을 서브프로세스로 재현해 임포트가 통과하는지까지 검증 — 마지막 두 테스트는 사용자가 직접 실행하다 겪은 실제 버그(TS-006)와 UI 재설계를 계기로 추가한 것이다). 여기에 더해 `prsti-qa-checklist`의 수동 절차를 병행 적용 중.

8. **배운 점**: "논문을 참고했다"는 것과 "논문에서 도출했다"는 것은 다르며, 이 차이를 항목 단위로 검증하지 않으면 지표의 학술적 근거가 조용히 흐려질 수 있음을 확인.
9. **회계·감사·IT 직무와의 연결**: 근거 없는 항목을 걸러내는 절차는 감사에서 "충분하고 적합한 감사증거" 개념과 본질적으로 같은 문제의식.
10. **STAR 소재**:
    - S: 기획안의 평가체계가 논문에서 실제로 도출됐는지 검증되지 않은 상태였다.
    - T: 19개 항목 전체를 논문 원문과 1:1로 대조해 출처를 확인해야 했다.
    - A: 표6~8을 항목 단위로 재추출하고, 기획안의 하위 항목과 대조해 불일치 3건을 발견했다.
    - R: 근거 없는 항목을 조용히 포함시키는 대신 별도 목록으로 분리하고, 논문 원 카테고리를 잠정 채택해 추적성을 확보했다.

## PE-002: 생성형 AI 근거 추출 컴포넌트의 환각 방지 설계

1. **해결하려던 문제**: 공시 원문에서 채점 근거를 뽑는 역할을 생성형 AI에 맡기면, AI가 원문에 없는 내용을 그럴듯하게 지어내는 환각(hallucination)이 시스템 신뢰성을 가장 크게 위협함.
2. **내가 수행한 구체적 행동**: evidence-extractor 에이전트와 prs-evidence-schema 스킬을 설계하면서 (1) 모든 `suggested_score>0`에는 원문 그대로의 `quote` 필드를 강제 (2) `quote`가 실제 원문(raw_excerpt)에 부분 문자열로 존재하는지 검증하는 절차를 prsti-auditor에 별도로 부여 (3) 근거를 못 찾으면 `not_found`로만 처리하고 추정 금지 (4) 신뢰도(confidence)를 "판정에 대한 확신"과 "존재 자체에 대한 확신"으로 구분해 혼동을 막음.
3. **사용한 기술과 선택 이유**: 구조화된 JSON 스키마(자유 서술형 대신) — 자유 서술은 검증이 불가능하지만, 스키마로 강제하면 기계적으로 대조 검증이 가능해짐.
4. **검토한 대안과 최종 의사결정**: AI가 직접 최종 점수를 내는 안(단순하지만 위험) vs AI는 제안만 하고 사람이 확정하는 안(복잡하지만 통제 가능) 중 후자를 채택 — 원 프로젝트의 핵심 원칙(AI/점수 분리)과 직결.
5. **테스트 및 검증 방법**: (설계 단계) prsti-qa-checklist 3절에 "quote가 raw_excerpt의 부분 문자열인지" 대조하는 절차를 명문화. 실제 코드 작성 시 이 절차를 자동화된 테스트로 전환 예정.
6. **정량적 결과**: 아직 실제 데이터로 측정된 수치 없음(설계만 완료, 코드 없음) — 개발 착수 후 환각 발생률을 측정할 계획.
7. **실패 또는 문제 해결 과정**: 해당 없음(설계 단계 사례).
8. **배운 점**: 생성형 AI를 신뢰할 수 있게 쓰는 방법은 "더 똑똑한 모델을 쓰는 것"이 아니라, AI 출력이 시스템의 최종 권위를 가지지 못하도록 구조로 막고 검증 가능한 형태(원문 인용)로만 받는 것이라는 점.
9. **회계·감사·IT 직무와의 연결**: "증거 없는 판정을 허용하지 않는다"는 원칙은 감사증거의 충분성·적합성 개념과 직접 연결됨.
10. **STAR 소재**:
    - S: 생성형 AI로 공시 근거를 자동 추출하는 컴포넌트가 필요했으나, AI 환각이 신뢰성을 해칠 위험이 컸다.
    - T: AI가 존재하지 않는 근거를 지어내지 못하도록 구조적으로 막는 설계가 필요했다.
    - A: 원문 인용 강제, 인용문 대조 검증 절차 분리, not_found 명시, 신뢰도 이원화를 스키마 수준에서 설계했다.
    - R: 실제 코드 작성 전 단계에서부터 환각 방지가 시스템 설계에 내장된 근거 추출 파이프라인을 완성했다.
