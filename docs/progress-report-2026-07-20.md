# PRSTI 진행상황 보고서 (2026-07-20)

이 문서는 지금까지 무엇을 했는지뿐 아니라 **왜 그렇게 했는지**를 설명합니다. 나중에 이 프로젝트를 다시 열어봤을 때(또는 다른 사람이 봤을 때) 판단 근거를 다시 추적할 수 있도록 남겨둡니다.

---

## 1. 지금까지 일어난 일의 큰 흐름

오늘 대화는 크게 두 단계로 진행되었습니다.

1. **1단계 — 자료 확인과 첫 방향 정리**: 저장소에 어떤 자료가 있는지 확인하고, 논문·기획초안·ARIX 기사를 바탕으로 프로젝트를 이해한 내용을 정리했습니다. 이 과정에서 **논문과 기획초안 사이에 구조적 불일치**를 하나 발견했습니다(3절에서 자세히 설명).
2. **2단계 — 하네스 구축과 Phase 0 실작업**: `/harness` 요청에 따라 앞으로 이 프로젝트를 진행할 "에이전트 팀 구조"를 만들었고, 그 구조를 실제로 써서 Phase 0(평가기준 초안)까지 진행했습니다.

아직 하지 않은 일: 배점 확정, 회계 판단기준 확정, 실제 채점 코드 작성, DART 공시 수집. 이것들은 원래 요청하신 대로 승인 없이 진행하지 않았습니다.

## 3. 왜 "5영역·100점" 기획안을 그대로 쓰지 않았는가

기획초안(그리고 이번 대화 처음의 지시문)에는 이런 5개 영역 구조가 제안되어 있었습니다.

> 거래 식별성(15점) / 계약 구조(25점) / 회계처리 투명성(20점) / 위험 정보(25점) / 정량성과 비교가능성(15점)

이 구조를 논문 원문(표6~8)과 하나하나 대조해봤는데, 대부분의 하위 항목은 논문에서 근거를 찾을 수 있었지만 **3개는 찾지 못했습니다**.

| 기획안의 하위 항목 | 문제 |
|---|---|
| **PRS 명칭** | 논문 4.2절에서 "기업마다 명칭이 다르다"는 **문제점**으로만 언급됩니다. 이걸 기업별로 0/1/2점 매기는 채점 항목으로 논문이 제안한 적은 없습니다. |
| **공시 위치** | 논문 5.2.3절은 "PRS 공시를 재무제표 주석의 정해진 위치에 모아야 한다"는 **지침 설계 원칙**을 말합니다. 이건 지침 자체가 지켜야 할 규칙이지, 개별 기업을 채점하는 항목이 아닙니다. |
| **진성매각 판단** | 논문 3.2.1절의 "위험과 보상의 이전" 논의와 관련은 있지만, 이건 표6의 항목6(회계처리 근거)에 딸린 **하위 판단 요소**이지 독립된 채점 항목으로 제안되지 않았습니다. |

**왜 이게 중요한가**: 이 프로젝트의 존재 이유는 "논문에서 제안한 공시 가이드라인을 정량화하는 것"입니다(처음 지시문 1절). 논문에 없는 채점 항목을 슬쩍 끼워 넣으면, PRSTI는 더 이상 "논문을 기반으로 한 지표"가 아니라 "논문을 참고했다고 주장하는, 실제로는 임의로 만든 체크리스트"가 됩니다. 그래서 이 3개 항목은 조용히 넣지 않고, `rubric.yaml`의 `excluded_candidates`라는 별도 섹션에 "왜 뺐는지"와 함께 남겨뒀습니다. 원하시면 다시 넣을 수 있고, 그 결정은 사용자님이 하셔야 한다고 판단했습니다.

**그래서 대신 무엇을 썼는가**: 논문이 원래 쓰고 있는 4개 카테고리(가.기본거래정보 / 나.정산및회계처리 / 다.투명성강화-부외부채 / 라.거래구조분류)에 조건부 필수·권장 항목을 별도 그룹으로 더해서 총 6개 그룹 구조를 임시로 채택했습니다. 이건 "이게 최종안이다"가 아니라 "추적성이 가장 확실한 안전한 기본값"입니다 — 5영역 재구성이 투자자 판단 순서상 더 낫다고 판단하시면 언제든 되돌릴 수 있습니다.

## 4. 하네스(에이전트 팀)를 만든 이유와 구조

### 4.1 왜 만들었는가

`/harness`를 요청하신 이유를 이렇게 이해했습니다: 이 프로젝트는 Phase 0(평가기준)부터 Phase 4(웹 서비스)까지 여러 단계를 거치는데, 매번 새로 대화를 시작할 때마다 "논문 내용이 뭐였지", "무엇을 조심해야 하지"를 처음부터 다시 설명하면 비효율적입니다. 그래서 **역할별 담당자(에이전트)와 그 담당자가 따라야 할 절차(스킬)를 파일로 저장**해두면, 나중에 어떤 세션에서 다시 열어도 같은 원칙을 지키면서 이어갈 수 있습니다.

### 4.2 원래 계획과 실제로 다르게 만든 부분

`/harness` 스킬은 기본적으로 "에이전트 팀"(여러 에이전트가 서로 실시간으로 메시지를 주고받으며 협업하는 방식)을 권장합니다. 그런데 확인해보니 **지금 이 환경(Claude Code / VSCode 확장)에는 그 방식에 필요한 도구(TeamCreate, TaskCreate)가 없었습니다.** 있는 것은 "제가 각 담당자를 하나씩 호출하고, 그 결과를 파일로 받아서 다음 담당자에게 넘겨주는" 방식뿐이었습니다(서브에이전트 방식이라고 부릅니다).

그래서 원래 권장되는 방식 대신, 실제로 쓸 수 있는 방식으로 설계를 바꿨습니다. 이건 임의로 기준을 낮춘 게 아니라 — 안 되는 도구를 쓰는 척하며 하네스를 만들면 나중에 실행할 때 전부 에러가 나기 때문에, 실제로 작동하는 구조를 만드는 게 우선이라고 판단했습니다.

### 4.3 담당자(에이전트) 5명

| 담당자 | 하는 일 | 왜 따로 분리했는가 |
|---|---|---|
| **rubric-architect** | 논문을 읽고 채점기준(rubric.yaml)을 만든다 | 회계 문헌 해석이라는 전문성이 다른 작업과 확실히 다름 |
| **dart-researcher** | DART에서 공시 원문을 찾아온다 | 외부 시스템(DART API) 연동이라는 별도 전문성 |
| **evidence-extractor** | 찾아온 원문에서 항목별 근거를 뽑아 "제안 점수"를 만든다 | AI가 점수를 뽑는 단계와, 그 점수를 확정하는 단계를 분리해야 한다는 게 처음 지시문의 핵심 원칙(6.1절)이었기 때문에, 아예 담당자를 다르게 둠 |
| **scoring-engine-dev** | rubric.yaml과 (사람이 확정한) 점수를 가지고 실제 최종 점수를 계산하는 코드를 만든다 | 마찬가지로 "점수를 확정하는 주체"를 AI와 분리하기 위해 |
| **prsti-auditor** | 위 네 담당자의 결과물이 서로 잘 맞아떨어지는지 검사한다 | 아래 4.4에서 자세히 설명 |

이 역할 분리는 처음 지시문의 6.1절("AI와 점수 계산을 분리한다")을 실제 작업 구조로 그대로 옮긴 것입니다 — evidence-extractor(AI)가 아무리 확신을 갖고 점수를 제안해도, 그 값은 항상 "제안값(suggested_score)"으로만 남고, scoring-engine-dev가 다루는 "확정값(confirmed_score)"에는 사람이 검토한 값만 들어갑니다.

### 4.4 검증 담당자(prsti-auditor)를 따로 둔 이유

일반적으로 "검증"이라고 하면 "파일이 잘 만들어졌나 확인하는 것"을 떠올리기 쉬운데, 실제로 문제가 되는 건 대부분 **두 산출물이 만나는 지점**입니다. 예를 들면:

- rubric-architect가 만든 항목 이름과, evidence-extractor가 그 항목에 붙이는 이름 표기가 미묘하게 다르면 → 나중에 점수 계산 단계에서 그 항목이 통째로 누락됩니다.
- evidence-extractor가 "이 문장이 원문에 있다"고 인용했는데 실제로는 dart-researcher가 가져온 원문에 그 문장이 없으면 → AI가 근거를 지어낸 것(환각)인데 겉보기엔 멀쩡해 보입니다.

이런 문제는 산출물 하나만 따로 봐서는 안 잡히고, **두 산출물을 동시에 펼쳐놓고 대조**해야 잡힙니다. 그래서 prsti-auditor에게는 "존재하는지 확인"이 아니라 "왼쪽과 오른쪽을 동시에 읽고 어긋난 지점을 찾으라"는 절차를 명시적으로 심어뒀고, 전체 작업이 다 끝난 뒤 한 번에 검사하는 대신 **각 단계가 끝날 때마다 바로바로 검사**하도록 설계했습니다(문제가 뒤로 갈수록 발견 비용이 커지기 때문입니다).

### 4.6 각 단계 전에 승인이 필요하도록 막아둔 것

처음 지시문에서 "승인 없이 하지 말라"고 하신 항목들을, 이번엔 제 기억에만 의존하지 않고 **에이전트 정의 파일 안에 규칙으로 박아뒀습니다.** 예를 들어 scoring-engine-dev 담당자 정의에는 이렇게 적혀 있습니다:

> "이 에이전트는 사용자가 Phase 0 산출물을 명시적으로 승인하기 전까지 실제 코드를 작성하지 않는다."

이렇게 해두면, 나중에 다른 대화에서 이 담당자를 불러도 (제가 이번 대화 내용을 기억하지 못하더라도) 같은 제약이 자동으로 적용됩니다.

## 5. Phase 0 실작업 — 무엇을 만들었고 왜 여기까지만 했는가

하네스를 만들고 나서, 승인이 필요 없는 부분(순수 문서화 작업)까지는 실제로 진행해뒀습니다.

### 5.1 `docs/scoring-methodology.md`

논문의 19개 항목(필수 12개 + 조건부 필수 3개 + 권장 4개) 하나하나에 대해:
- 0점/1점/2점이 각각 무엇을 의미하는지
- 실제로 이 정도면 2점이다 싶은 논문 속 예시(있는 경우만 — 없으면 억지로 안 만들고 비워둠)
- 논문 표10에서 확인된, 이 항목을 공시하지 않은 기업 사례(반례)

를 정리했습니다. 이 과정에서 알게 된 흥미로운(그리고 우려되는) 사실이 있습니다: **항목 7(부외부채 재분류 위험)·8(부채 분류시 영향액)·11(콜옵션 조건)·12(재매입 조건)은 논문이 조사한 10개 기업 전원이 공시하지 않았습니다.** 이 4개 항목은 앞으로 실제 기업을 채점하면 거의 항상 0점이 나올 가능성이 큽니다 — 나쁜 일은 아니지만(오히려 논문이 지적한 문제를 정확히 보여주는 것), 나중에 "이 지표가 기업 간 차이를 잘 구분하는가"를 검증할 때 이 4개 항목이 구분력에 기여를 못 할 수 있다는 점을 미리 적어뒀습니다.

### 5.2 `rubric/rubric.yaml` (초안)

위 문서를 사람이 읽는 버전이라면, 이건 나중에 Python 코드가 읽을 "기계용" 버전입니다. **모든 배점(weight) 칸은 비워뒀습니다(null)** — 배점을 정하는 건 회계적 판단이 필요한 일이라 처음 지시문에서 승인 필요 항목으로 명시하셨기 때문입니다. 상태(`status`)도 `"draft"`(초안)로 표시해서, 나중에 코드가 이 파일을 읽을 때 "아직 확정 아님"을 자동으로 알 수 있게 했습니다.

### 5.3 `_workspace/rubric_gaps.md`

작업하면서 "이건 제가 임의로 못 정하겠다" 싶은 지점 6가지를 정리했습니다. 예를 들면:

- 논문이 "가격보전조건이 없다"고 명시하는 것도 유효한 공시로 인정하는데, 이걸 채점에서 어떻게 다뤄야 할지(0점과 똑같이 취급하면 억울한 감점이 됩니다)
- 신용평가사 같은 제3자가 분석한 자료를 "기업의 공시"로 인정할지, 아니면 기업 스스로 밝힌 것만 인정할지

이런 것들은 회계적 판단이나 프로젝트 방향에 대한 결정이라 제가 넘겨짚지 않고 질문으로 남겨뒀습니다.

## 6. 아직 안 한 일 (의도적으로)
