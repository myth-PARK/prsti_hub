## TS-001: 논문 MD 추출본이 부록 사례 중간에서 끊김

- **일자**: 2026-07-20
- **증상**: 논문 원문 MD 파일을 끝까지 읽었을 때, 부록 "사례별분석"이 10개 기업 중 4개(한화솔루션·SK·LG화학·에코프로)까지만 서술되고 에코프로 사례 문장 중간("...공시의 투명성에는")에서 파일이 끝남.
- **원인 분석**: `Get-Content`로 총 라인 수(594줄)를 확인해 파일 자체가 그 지점에서 물리적으로 끝난다는 것을 확인. PDF 원본이 있으나 이 환경에 `poppler`(PDF 렌더링 도구)가 설치되어 있지 않아 PDF에서 직접 재추출 불가.
- **시도했으나 실패한 방법**: `pypdf`/`PyPDF2` Python 모듈로 PDF 텍스트 추출 시도 → 두 모듈 모두 미설치로 실패. `Read` 도구로 PDF 페이지 렌더링 시도 → `pdftoppm`(poppler) 미설치로 실패.
- **최종 해결(1차, 2026-07-20)**: 논문 본문의 표2~4(10개 기업 요약 비교)와 표10(10개 기업 × 12개 필수항목 공시여부)이 이미 10개 기업 전체를 다루고 있어, 이를 근거로 Phase 0을 진행하기로 함. 누락된 6개 기업의 부록 서사는 `_workspace/rubric_gaps.md`에 보완 필요 항목으로 기록.
- **최종 해결(2차, 2026-07-21 — 완전 해결)**: 사용자가 원본 PDF(`회계저널 6월호 - 추출본_박신화.pdf`) 경로를 다시 제시하며 보완을 요청. 이 시점에는 `pypdf`·`pdfplumber`가 이미 설치되어 있음을 확인(1차 시도 때와 달리 poppler 없이도 pdfplumber로 추출 가능)하고, 50페이지(=논문 p.280, 부록 끝) 전체를 재추출. 기존 MD가 끊긴 지점("...공시의 투명성에는")부터 정확히 이어서 5~10번(효성화학·롯데케미칼·CJ ENM·넷마블·두산·금호건설) 서사를 원문 그대로(PDF 줄바꿈만 문단으로 재구성) 보충.
- **재발 방지 조치**: `docs/troubleshooting-log.md`에 "이 환경은 시간이 지나면서 poppler 없이도 pdfplumber만으로 PDF 텍스트 추출이 가능해질 수 있다"는 사실을 기록 — 향후 유사 블로커 발생 시 도구 재설치 여부를 먼저 확인할 것.
- **관련 development-log**: DEV-20260720-01, DEV-20260721-01
- **상태**: **완전 해결** — `.claude/skills/prs-rubric-traceability/references/paper-items.md`의 "부록 사례 커버리지" 절 갱신 완료

## TS-002: 하네스 기본 실행 모드(에이전트 팀)에 필요한 도구 부재

- **일자**: 2026-07-20
- **증상**: `/harness` 스킬 지침대로 에이전트 팀(TeamCreate 기반) 구조를 설계하려 했으나, 이 세션에서 사용 가능한 도구 목록에 TeamCreate/TaskCreate가 보이지 않음.
- **원인 분석**: `ToolSearch`로 "TeamCreate TaskCreate team task management" 및 `select:TeamCreate,TaskCreate,TaskUpdate,...` 쿼리를 실행한 결과 "No matching deferred tools found" — 이 환경 자체에 해당 기능이 배포되어 있지 않음을 확인.
- **시도했으나 실패한 방법**: 없음(도구가 아예 없으므로 재시도 대신 대안 설계로 즉시 전환)
- **최종 해결**: `Agent` 도구(서브에이전트 스폰) + `SendMessage`(후속 대화) 조합으로 파이프라인을 재설계. `prsti-harness` 스킬 상단에 "실행 모드: 서브 에이전트"와 그 이유를 명시해, 향후 세션에서 같은 문제를 다시 조사하지 않도록 함.
- **재발 방지 조치**: `prsti-harness` skill.md에 이 제약과 대안을 문서화 완료 → DEC-001로 연결
- **관련 development-log**: DEV-20260720-02
- **상태**: 해결

## TS-003: git 명령어를 찾을 수 없음

- **일자**: 2026-07-20
- **증상**: PRSTI 폴더에서 `git status` 실행 시 "git : The term 'git' is not recognized..." 에러.
- **원인 분석**: 이 Windows 환경 PATH에 git이 설치되어 있지 않음.
- **시도했으나 실패한 방법**: `Test-Path .git` 확인(디렉토리 자체도 없음을 확인), `git -C ... status` 직접 실행(실패).
- **최종 해결**: 임의로 git을 설치하거나 초기화하지 않고, 커밋 관련 필드는 모두 "N/A — git 미도입"으로 통일 처리하기로 함 → DEC-004로 연결.
- **재발 방지 조치**: 없음(사용자 결정 대기 사안)
- **관련 development-log**: DEV-20260720-01
- **상태**: 미해결(보류 — 사용자 결정 필요)

## TS-004: rule_based_extractor가 문서 목록의 첫 매치만 채택해 더 강한 증거를 놓침

- **일자**: 2026-07-21
- **증상**: 한화솔루션 29건 실 데이터에 규칙을 적용한 첫 실행에서 필수-05(정산방식)가 1점으로 나왔다. 그런데 29건 중 뒤쪽 문서에 "정산기준금액=(매각 완료 기초자산의 총 매각 금액-매각 관련 비용)-매각 완료 기초자산에 해당하는 계약금액"이라는, 2점 판정 근거로 정의해 둔 `formula_expression` 패턴에 정확히 매치되는 문장이 있었다.
- **원인 분석**: `pipeline.py`의 후보 선택 로직이 "documents 리스트를 순회하다 처음 매치되는 것을 채택하고, negative_keyword나 na_expression 매치가 있을 때만 나중 문서로 교체"하도록 돼 있었다. 즉 리스트 앞쪽에 있는 약한 키워드 단독 매치(예: "정산"이라는 단어만 포함된 문서)가 뒤쪽의 더 구체적인 패턴 매치(formula_expression까지 만족하는 문서)를 가리는 구조였다.
- **최종 해결**: 선택 기준을 "리스트 순서상 첫 매치"에서 "매치된 patterns 개수가 더 많은 후보 우선"으로 변경(negative_keyword·na_expression 매치는 여전히 발견 즉시 최우선 채택). 수정 후 재실행 결과 필수-05가 2점으로 바뀌었고(총점 25.5→30.75), 나머지 항목은 영향 없음을 회귀 테스트로 확인.
- **재발 방지 조치**: `tests/test_rule_based_extractor.py`에 이 로직을 직접 검증하는 테스트는 아직 추가하지 않음(회귀 테스트는 confirmed 항목 위주로만 있음) — 향후 여러 문서에 걸친 우선순위 선택 로직을 추가로 변경할 경우, `_workspace/derived/scored_한화솔루션_2025.json`을 재실행해 필수-05 점수가 유지되는지 수동 확인 필요.
- **관련 development-log**: DEV-20260721-03
- **상태**: 해결
